  Strengths of this approach:
                                                                                                                                           
  - Problem fit: Batched tensor-times-vector (TTV) is mathematically simple (just batched mat-vec / dot products), fully controllable by tensor
  dimensions I, J, K, and will show clear GPU speedup — exactly what's asked for.
  - Serial simplification is correct: Replacing BLAS calls with naive triple loops (like the multiplyMatrices style in MatrixMult.c) makes the serial
   version pedagogically transparent and a fair baseline.
  - MEX removal is natural: The MEX interface is just a MATLAB wrapper. Stripping it down to a standalone main() with command-line dimension
  arguments (like ./batched_ttv I J K) is a clean transformation.
  - Scientific motivation is real: You have a genuine research use case (CP decomposition / tensor methods), which satisfies the "Why" slide in the
  presentation.

  Things to plan for to satisfy all requirements:

  - Performance optimization beyond naive: The naive CUDA port would be one kernel per batch element. For the optimization, shared memory tiling of

  case (m, n combination), which gives you something interesting to discuss.

  - Pick a subset of the (m, n) cases: The current code handles 6 cases. For the project, you could focus on one or two (e.g., m=1, n=3 and m=2, n=3)
   to keep scope manageable.

  One potential risk: The 6 different memory-layout cases in batched_ttv.c each require a different CUDA kernel or branching logic. I'd recommend
