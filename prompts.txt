⏺ Your concern is valid but I think the framing resolves it. Let me walk through the arithmetic:

  What the benchmark actually measures

  X is generated in GPU VRAM (not loaded from disk). So the relevant comparison is:
  - Serial CPU: reads X from DDR4 RAM (~50 GB/s bandwidth)
  - Parallel GPU: reads X from GPU VRAM (~500-900 GB/s bandwidth)

  Disk I/O never enters the picture. The speedup comes from memory bandwidth, not compute.

  This algorithm is memory-bandwidth-bound — which is fine

  Arithmetic intensity for one output element Y[k,i]:
  - Load J floats from X, J floats from U → ~8J bytes
  - Do 2J FLOPs (multiply-accumulate)
  - AI ≈ 0.25 FLOP/byte

  Low AI means it's bandwidth-bound, but GPU bandwidth advantage over CPU RAM is 10-20x. That's a real and meaningful speedup.
   "Memory-bandwidth-bound operations benefit from GPU's high-bandwidth memory" is a perfectly valid HPC story.

  What this means for scope

  Scaling beyond VRAM doesn't make sense for this project — once you're streaming through PCIe (~16 GB/s), you're slower than
  CPU RAM. The interesting regime is entirely within VRAM. Your 1000×1000×100 case works; just benchmark a sweep of sizes that
   fit (e.g., varying K from 10 to 100, varying I/J from 100 to 1000) and show the speedup curve.

  One flag worth knowing

  The current kernel likely has poor memory coalescing: consecutive threads (varying k, your x-dim) access X[i + j*I + k*I*J]
  with stride I*J between them — that's up to 4MB between adjacent threads in a warp, which will hurt measured throughput
  significantly. This might be worth fixing before benchmarking if you want results that reflect the algorithm's true
  potential rather than the coalescing penalty.

  The project is fine — you just don't need to scale beyond VRAM.

  ---                                                                                                                         
  Why some cases are slower than others                                                                                       
                                                                                                                              
  The tensor X is stored column-major: X[i + j*I + k*I*J]. The innermost summation loop accesses X with different strides
  depending on which mode is being contracted:                                                                                
  
  ┌────────────┬──────────────────┬──────────────┬─────────────────────────────┐                                              
  │    Case    │ Inner loop index │   X stride   │             Why             │                                            
  ├────────────┼──────────────────┼──────────────┼─────────────────────────────┤                                              
  │ m1n3, m1n2 │ i                │ 1 (stride-1) │ Sequential — cache loves it │                                            
  ├────────────┼──────────────────┼──────────────┼─────────────────────────────┤
  │ m2n3, m3n2 │ j or k           │ I = 1000     │ 4KB jumps                   │                                              
  ├────────────┼──────────────────┼──────────────┼─────────────────────────────┤                                              
  │ m3n1       │ k                │ I×J = 10⁶    │ 4MB jumps — cache disaster  │                                              
  └────────────┴──────────────────┴──────────────┴─────────────────────────────┘                                              
                                                                                                                            
  That's why m3n1 takes ~10s and m1n3 takes ~3.3s, even though both do exactly I×J×K = 1B multiply-adds.                      
                                                                                                                            
  ---                                                                                                                         
  OpenMP scaling                                                                                                            
                
  The cache-friendly cases (m1n3–m3n2) scale almost perfectly — at 32 threads you get ~25x speedup. The m2n1/m3n1 cases only
  get ~3x at 32 threads because they're bandwidth-bottlenecked: more threads just fight harder over the same memory bus.      
  
  ---                                                                                                                         
  Why the GPU obliterates OpenMP                                                                                            
                                
  The GPU timing (0.019s) only covers the kernel + D2H copy — the H2D transfer of the 4GB tensor is not timed (look at where
  cudaEventRecord(start) is called, line 121, after the memcpys). So you're comparing GPU kernel execution to full CPU wall   
  time, which gives you an even more dramatic number. Even accounting for that, the GPU wins because:
                                                                                                                              
  1. It dispatches ~1M threads (one per (i,k) output element) all at once vs 32 CPU threads                                   
  2. GPU HBM bandwidth is ~10–20x higher than CPU DRAM bandwidth
                                                                                                                              
  ---                                                                                                                       
  Why 2k is painfully slow
                                                                                                                              
  The work scales as N³. Going from 1k to 2k:
                                                                                                                              
  - Data: 1k tensor = 4 GB → 2k tensor = 32 GB (8×)                                                                           
  - Work: 1B ops → 8B ops (8×)
                                                                                                                              
  So each case takes ~8× longer. Your 1k serial run took ~30s total for all 6 cases. At 2k that's ~4 minutes just for serial, 
  then 6 OpenMP thread counts × ~2–6 min each = easily 30–45 minutes for the whole benchmark.

  observing is the serial/OpenMP portion of the benchmark running before the GPU step.
